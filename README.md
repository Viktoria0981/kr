# Инструкция: автоматическое подключение библиотеки

## Как сделать, чтобы библиотека "зарегистрировала" себя в системе вашего пользователя

### Шаг 1: Подготовка библиотеки (добавление «паспорта»)

Откройте `CMakeLists.txt` проекта библиотеки. Добавьте инструкции по установке в самый конец файла.

**Важно:** Посмотрите, как называется цель библиотеки в команде `add_library(...)`. Допустим, она называется `my_math_lib`.

Добавьте этот код (замените `my_math_lib` на реальное название):

```cmake
# 1. Говорим, куда копировать файлы при установке
install(TARGETS my_math_lib 
        EXPORT MyLibTargets            # Метка для экспорта
        LIBRARY DESTINATION lib        # Библиотеки (.so) -> в папку lib
        ARCHIVE DESTINATION lib        # Статические библиотеки -> в папку lib
        INCLUDES DESTINATION include)  # Где искать заголовки

# 2. Копируем заголовочные файлы (замените на свои .h файлы)
install(FILES my_math_lib.h DESTINATION include)

# 3. Генерируем конфиг-файл (тот самый "паспорт")
# Чтобы find_package(MyLib) в приложении сработал
install(EXPORT MyLibTargets
        FILE MyLibConfig.cmake
        NAMESPACE MyLib::
        DESTINATION lib/cmake/MyLib)

Шаг 2: Установка библиотеки в локальную директорию
Теперь нужно «установить» библиотеку в вашу домашнюю папку (~/.local), чтобы не использовать sudo. Выполните в терминале, находясь в папке библиотеки:

bash
# 1. Конфигурируем сборку и указываем путь установки в ~/.local
cmake -B build -DCMAKE_INSTALL_PREFIX=$HOME/.local

# 2. Компилируем
cmake --build build

# 3. Устанавливаем (файлы разлетятся по папкам в ~/.local)
cmake --install build
Шаг 3: Настройка "автозапуска" для системы (через .bashrc)
Чтобы приложение видело библиотеку при сборке и находило её при запуске без лишних команд, настройте ваш профиль пользователя.

Откройте файл конфигурации:

bash
nano ~/.bashrc
Пролистайте в самый низ и вставьте эти две строки:

bash
# Чтобы CMake находил установленные конфиги библиотек
export CMAKE_PREFIX_PATH="$HOME/.local:$CMAKE_PREFIX_PATH"

# Чтобы приложение находило динамическую библиотеку (.so) при запуске
export LD_LIBRARY_PATH="$HOME/.local/lib:$LD_LIBRARY_PATH"
Сохраните и выйдите: Ctrl+O → Enter → Ctrl+X

Примените настройки прямо сейчас:

bash
source ~/.bashrc
Шаг 4: Сборка и запуск приложения
Теперь вы можете идти в папку приложения. Вам не нужно ничего менять в его файлах. Просто запускайте стандартные команды:

bash
# Переходим в папку приложения
cd /path/to/app_source

# 1. Конфигурируем (теперь find_package сработает автоматически!)
cmake .

# 2. Собираем
cmake --build .

# 3. Запускаем
./имя_твоего_приложения
Почему это сработает после релогина?
Файлы библиотеки физически лежат в вашем $HOME/.local

Переменные CMAKE_PREFIX_PATH и LD_LIBRARY_PATH прописаны в .bashrc, а этот файл выполняется автоматически каждый раз, когда вы открываете терминал или входите в систему

Приложение при сборке через find_package() заглянет в CMAKE_PREFIX_PATH, увидит там ваш MyLibConfig.cmake и узнает, где лежат заголовки и сама библиотека```

Markdown
# Решение задачи №1: Локальная линковка библиотеки

Данное решение демонстрирует способ подключения внешней библиотеки к приложению **без правки исходного кода** (как библиотеки, так и приложения) и **без использования прав администратора (sudo)**.

---

## 1. Сборка и подготовка библиотеки
Если исходный код библиотеки закрыт для редактирования, мы собираем её "как есть" и вручную организуем структуру файлов в домашней директории.

```bash
# В директории с исходным кодом библиотеки:
cmake -B build
cmake --build build

# Создаем структуру папок в домашнем каталоге
mkdir -p $HOME/.local/lib
mkdir -p $HOME/.local/include

# Копируем артефакты сборки
cp build/*.so $HOME/.local/lib/
cp *.h $HOME/.local/include/
2. Настройка окружения пользователя
Для того чтобы инструменты сборки (компилятор и линковщик) автоматически находили файлы в ~/.local, необходимо настроить переменные окружения в файле ~/.bashrc.

Откройте файл: nano ~/.bashrc

Добавьте в конец следующие строки:

Bash
# Путь для поиска заголовочных файлов (.h) компилятором
export CPATH="$HOME/.local/include:$CPATH"

# Путь для поиска библиотек (.so) линковщиком при сборке
export LIBRARY_PATH="$HOME/.local/lib:$LIBRARY_PATH"

# Путь для поиска библиотек динамическим загрузчиком при запуске программы
export LD_LIBRARY_PATH="$HOME/.local/lib:$LD_LIBRARY_PATH"
Обновите текущую сессию терминала:

Bash
source ~/.bashrc
3. Сборка приложения
Теперь приложение собирается стандартной командой. Благодаря настроенному окружению, CMake успешно завершит процесс, даже если find_package не найдет специфические конфигурационные файлы библиотеки.

Bash
# В папке, где вы хотите собрать приложение (например, build_app):
cmake /path/to/app/source && cmake --build .
Техническое описание решения (для защиты)
Как это работает?
Поиск заголовков: Переменная CPATH заставляет gcc/g++ искать заголовочные файлы в ~/.local/include так же, как если бы они лежали в /usr/include.

Линковка по имени: Когда find_package в CMakeLists.txt приложения не находит файл конфигурации (Config.cmake), CMake (согласно условию задачи) пробует слинковать библиотеку по её имени (например, -lmath_lib). Линковщик находит нужный файл .so в папке, указанной в LIBRARY_PATH.

Запуск: Чтобы скомпилированный файл запустился, ОС должна знать, где лежит динамическая библиотека. За это отвечает LD_LIBRARY_PATH.

Почему это лучшее решение в данном контексте?
Соблюдена чистота кода: Мы не внесли ни одной правки в CMakeLists.txt проектов.

Портативность: Решение будет работать на любом Linux-сервере в рамках прав текущего пользователя.

Автоматизация: После однократной настройки .bashrc все последующие сборки и запуски происходят автоматически без ввода дополнительных путей.

Структура файлов в $HOME/.local
После выполнения всех шагов структура должна выглядеть так:

Plaintext
.local/
├── include/
│   └── math_lib.h      # Заголовочный файл
└── lib/
    └── libmath_lib.so  # Динамическая библиотека

### Что еще стоит проверить перед сдачей:
* **Имена файлов:** Если твоя библиотека называется не `libmath_lib.so`, а как-то иначе, убедись, что ты копируешь именно нужный файл (обычно это `*.so`).
* **Путь к приложению:** В команде сборки замени `/path/to/app/source` на реальный путь к папке с приложением на твоем компьютере.
